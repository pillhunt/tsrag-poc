## **Что PoC делает в целом**

1. Вы вводите **описание** и **путь к логам**.
2. **Шаг 0 (LLM)** вытаскивает дату, время, симптомы, шаблоны для поиска по времени и ключевые слова.
3. **Шаги 1–2** режут логи по времени в один список строк.
4. **Шаги 3–5** ищут по этому списку: ключевые слова, долгие HTTP, ошибки.
5. **Шаг 6 (LLM)** пишет заключение по **сводке** шагов 0–5, не по полным логам.

Ниже — что **не подходит для любого инцидента на любой системе**, если ничего не менять.

---

## **A. Привязка к вашему компьютеру и папкам**

1. **По умолчанию логи лежат на** `D:\RAG`, в Docker это `/rag`. Другой диск или путь — нужно править `docker-compose` или переменные окружения.
2. **По умолчанию caseone** — `D:\RAG\tsrag\temp\uploads\caseone` → `/caseone`. Это ваш путь, не «любой проект».
3. **В health и подсказках** ищутся папки, начинающиеся с `REN`. Сервер с именем `APP-SRV_2026-05-01` в подсказках не появится.
4. **Если в пути есть** `ren-mskcaspro01`, PoC подсказывает папки вида `REN-MSKCASPRO01` / `REN-MSKCASPRO01_2026-04-23`. Это под **один** известный сервер, не под любой хост.
5. **Ollama** ожидается в Docker-сети `shared-network`, контейнер `tsrag-ollama`. На другом стенде без этого compose PoC к модели сам не подключится.

**Итог:** PoC заточен под ваш layout диска и ваш Docker, не под «скопировал на любой сервер и работает».

---

## **B. Привязка к продукту CaseOne / CaseMap**

1. PoC рассчитан на **логи CaseOne**: middleware, `global.log`, `WorkflowTrace.log`, `ClientLogs.log`. Другой продукт (1С, nginx-only, K8s) — эти имена и форматы **не подхватятся сами**.
2. **Долгие запросы (шаг 4)** читаются только из логов, в имени которых есть `RequestLoggingMiddleware`, и только строки вида `GET/POST … /api/... … миллисекунды`. Другой формат лога → шаг 4 будет пустым, даже если запросы были долгими.
3. **Ошибки (шаг 5)** по умолчанию ищутся **только в файлах, в пути которых есть** `global.log`. Ошибки в других файлах среза **игнорируются**, если не менять настройку API.
4. **Типы ошибок** заданы списком в коде: deadlock, дубликат PK, timeout, «карточка изменилась», обрыв соединения. Текст вне этого списка попадёт в «общая ошибка» или не попадёт вообще.
5. В промптах шагов 0 и 6 прямо написано: инциденты **CaseOne/CaseMap**. Модель настроена на этот домен.
6. Поле `caseone_path` **в форме проверяется** (есть ли папка) и **передаётся в LLM на шаге 0**. **Ни один шаг 1–6 по файлам из caseone не ходит.** Код конфигурации и API-док из caseone **не читаются**.

**Итог:** PoC — разбор **логов CaseOne**, а не «любой системы». Caseone в форме сейчас почти декоративный.

## **C. Привязка к одному сценарию («кнопка», сохранение, ProjectTypes)**

1. В **примере в форме** — текст про сохранение новой кнопки и 20–30 минут. Это не ломает код, но подталкивает к одному кейсу.
2. ~~В **промпте шага 0** в примерах ключевых слов: `save`, `button`, `PutProjectType`.~~ **Исправлено:** примеры в `intent_table_system.md` нейтральные (таймаут, отчёт, `/api/`).
3. Если LLM не вернула цель разбора, в коде подставляется цель про **«долгое сохранение и ошибки сервера»** — снова сценарий «save».
4. ~~В **README** все примеры — `REN-MSKCASPRO01_2026-04-23`, сохранение, `PutProjectType`.~~ **Частично:** API-примеры в README нейтрализованы; блок Docker про `REN-*` остаётся про монтирование томов.
5. **Классификации** `ProjectTypes` **vs** `Projects` **в коде нет.** Есть только группировка по полному URL в шаге 4. Отдельно «это сохранение типа проекта, а не проекта» PoC **не определяет**.

**Итог:** PoC не зашит на один тикет в коде шагов 3–5, но **шаг 0 и UI смещены** под историю с кнопкой и сохранением.

## **D. Жёсткие числа (не из жалобы пользователя)**

1. **Медленный запрос** = не короче **5 минут** (300 000 мс). Зашито в кнопке шага 4 в UI. Инцидент «тормозит 30 секунд» на шаге 4 **ничего не покажет**, пока не измените порог в запросе API вручную.
2. **Связка ошибки с долгим запросом** — окно **±90 секунд**. Зашито в UI шага 5.
3. В срез попадает **не больше 100 000 строк**; остальное отбрасывается, в ответе флаг «срез обрезан». Шаги 3–5 видят **только то, что влезло**, не весь инцидент на диске.
4. Одна строка лога в сэмплах обрезается до **2000 символов**.
5. В досье для шага 6 — ограниченное число коротких цитат (порядка **20 фрагментов по ~500 символов**). Это не полные логи.

**Итог:** Пороги «медленно» и «рядом по времени» — **фиксированные**, не из описания инцидента.

## **E. Что задумывали в алгоритме, но в PoC нет**

1. **Парсинга** `WorkflowTrace.log` **нет** — файл может попасть в срез по времени, но отдельного шага «этапы workflow, завис на save» **нет**.
2. **Парсинга client / console логов нет** — нельзя вывести «не браузер» / «обрыв сети на клиенте» из кода PoC.
3. **Исходный код и API-документ не используются** — только текст жалобы и логи.
4. **HITL** (правка заключения человеком) **нет**.

**Итог:** Вывод шага 6 про клиент, браузер и смысл операции **ограничен** тем, что есть в middleware + global + keywords. Без trace/client это **не доказывается**, только предполагается или попадает в «не подтверждено».

---

## **F. Фильтр по времени (частая ловушка)**

1. Время в логах ищется **подстрокой**, например `2026-04-23 20:`. Если в файле другой формат (`23.04.2026`, UTC без даты, только время) — **срез будет пустым**, хотя логи на диске есть.
2. Часовой пояс **нигде явно не задаётся**. «8 вечера» в промпте переводится в 20:00, но **не привязывается к TZ сервера логов**.

**Итог:** Универсальность по времени зависит от того, совпадает ли формат меток в логах с тем, что сгенерировал шаг 0.

---

## **G. Что универсально (работает для другого инцидента CaseOne на тех же логах)**

1. **Текст жалобы**, **путь к логам**, **дата из имени папки** (если есть `YYYY-MM-DD`) — не привязаны к REN.
2. **Окно времени, patterns, keywords** — из шага 0 (LLM + пользователь), не захардкожены в Python для шага 3.
3. **Шаги 3–5** не ходят на диск повторно — только по срезу из 1–2 (если срез не пустой и не обрезан критично).
4. **Шаг 6** не читает логи с диска — только JSON сводки; другой инцидент = другие факты в сводке, если шаги 0–5 прогнаны.

**Итог:** Для **другого инцидента на том же типе логов CaseOne** (другая дата, другие слова, другой URL) PoC **нормально переиспользуется**, если поправить пути и не полагаться на примеры в промпте.

---

## **H. Ошибка в коде (не про инцидент, но вводит в заблуждение)**

1. В тексте «ошибок не найдено» всегда пишется **«только global.log»**, потому что в коде условие `if True` — всегда одна ветка. На самом деле режим зависит от флага запроса, но сообщение **не переключается**.

---

## **Одна фраза без двусмысленности**

- **Не универсально:** любой продукт, любой формат логов, любой сервер, любой порог «медленно», доказательство «не браузер», caseone/API/код, папки не `D:\RAG` / не `REN`.
- **Универсально в рамках CaseOne:** новая жалоба → шаг 0 → срез → поиск → заключение, если логи того же типа и формат времени совпадает.
- **Смещено под ваш пример:** UI, README, промпт шага 0, подсказки REN-MSKCASPRO01, fallback «сохранение», порог 5 минут и 90 секунд в UI.

Если нужно, следующим сообщением могу дать **короткий чеклист «что поменять в коде»** в том же формате — по одному пункту на файл.
