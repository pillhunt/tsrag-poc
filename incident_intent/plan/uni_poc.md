## **Что PoC делает в целом**

1. Вы вводите **описание** (и при необходимости **загружаете логи** в `temp/incidents/<id>/`).
2. **Шаг 0 (LLM)** вытаскивает дату, время, симптомы, шаблоны для поиска по времени и ключевые слова.
3. **Шаги 1–2** режут логи по времени в один список строк.
4. **Шаги 3–5** ищут по этому списку: ключевые слова, долгие HTTP, ошибки.
5. **Шаг 6 (LLM)** пишет заключение по **сводке** шагов 0–5, не по полным логам.

Ниже — что **не подходит для любого инцидента на любой системе**, если ничего не менять.

## **D. Жёсткие числа (не из жалобы пользователя)**

1. ~~**Медленный запрос** = 5 мин (300 000 мс) в UI.~~ **Частично исправлено:** порог из шага 0 (`min_slow_request_ms`, `duration_utils.py`); UI берёт значение из таблицы, fallback **60 000 мс** (`POC_DEFAULT_MIN_SLOW_MS`). *Остаток:* дефолт API `slow_requests` = 300 000, если вызвать без таблицы.
2. **Связка ошибки с долгим запросом** — окно **±90 секунд** — **не исправлено** (дефолт API и UI шага 5).
3. **Лимит среза 100 000 строк** — **не исправлено** (`max_time_window_lines`); флаг `time_window_truncated` работает.
4. **Обрезка строки до 2000 символов** в шагах 3–5 — **не исправлено** (намеренный лимит).
5. **Досье шага 6: ~20 фрагментов × ~500 символов** — **не исправлено** (`evidence_bundle.py`).

**Дополнительно (улучшение, не в исходном списке):** второе окно `slow_time_window_lines` (±`POC_SLOW_WINDOW_PADDING_H`, по умолчанию 1 ч) для шагов 4–5; расширение «одного времени» ±`POC_TIME_PADDING_MIN` для жалобы.

**Итог:** порог «медленно» в основном из жалобы; ±90 с, лимиты среза и цитат — по-прежнему фиксированы.

## **G. Что универсально (работает для другого инцидента без переписывания кода)**

1. **Чат + upload**, **temp/incidents**, **дата из имени папки** (`YYYY-MM-DD`) — **да**, не привязаны к REN.
2. **Окно времени, patterns, keywords, порог slow** — из шага 0 (LLM + `duration_utils`), не захардкожены для шага 3.
3. **Шаги 3–5** — только срез из 1–2, без повторного чтения диска — **да**.
4. **Шаг 6** — только JSON-сводка — **да**.
5. **Многоформатные логи** (nginx, IIS, SQL, CaseOne middleware) — **да** (пункт B): discovery, access-парсеры, `error_rules.yaml`.
6. **Два временных окна** — жалоба (шаг 3, 6) и расширенное для slow/errors (шаги 4–5) — **да**.

**Итог:** новый инцидент = новая жалоба + upload + прогон шагов 0–6; ограничения — формат **времени** в срезе (F) и лимиты (D).
