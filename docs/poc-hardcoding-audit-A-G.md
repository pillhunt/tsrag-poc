# PoC: что привязано к одному инциденту и не универсально

Дата разбора: 2026-05-23 (по диалогу разработки PoC).

Цель документа — зафиксировать, **где PoC не подходит «для любого инцидента на любой системе»**, без двусмысленности. Формулировки — простым языком, как в исходном разборе по пунктам A–G.

---

## Что PoC делает в целом

1. Пользователь описывает инцидент (сейчас — в чате, с опциональной загрузкой логов).
2. **Шаг 0 (LLM)** извлекает дату, время, симптомы, шаблоны поиска по времени и ключевые слова (`search_keywords`).
3. **Шаги 1–2** режут логи по времени в один список строк (`time_window_lines`).
4. **Шаги 3–5** ищут по этому списку: ключевые слова, долгие HTTP, ошибки.
5. **Шаг 6 (LLM)** пишет заключение по **сводке** шагов 0–5, не по полным логам.

Ниже — что **не подходит для любого инцидента на любой системе**, если ничего не менять.

---

## A. Привязка к вашему компьютеру и папкам

1. **По умолчанию логи лежат на `D:\RAG`**, в Docker это `/rag`. Другой диск или путь — нужно править `docker-compose` или переменные окружения (`POC_RAG_HOST_PREFIX`, `POC_PATH_MAP`).
2. **По умолчанию caseone** — `D:\RAG\tsrag\temp\uploads\caseone` → `/caseone` (в `path_resolve.py`). Это путь вашего стенда, не «любой проект». *Частично снято:* в новом UX caseone берётся из `temp/caseone` в корне PoC (`POC_TEMP_DIR`), но старые маппинги в `path_resolve.py` остаются для обратной совместимости.
3. **В health и подсказках** (функция `path_hints_for_missing` в `path_resolve.py`) ищутся папки с префиксом **`REN`**. Сервер с именем `APP-SRV_2026-05-01` в подсказках не появится.
4. ~~**Если в пути есть `ren-mskcaspro01`**, PoC подсказывает папки вида `REN-MSKCASPRO01`.~~ **Исправлено:** `path_hints_for_missing()` перечисляет любые подкаталоги в `logs/`.
5. **Ollama** ожидается в Docker-сети `shared-network`, контейнер `tsrag-ollama`. На другом стенде без этого compose PoC к модели сам не подключится.

**Итог:** PoC заточен под ваш layout диска и ваш Docker, не под «скопировал на любой сервер и работает».

**Что уже сделано по пункту A (план «диалог + temp»):**

- Убраны поля ввода путей на странице; логи — `temp/incidents/<incident_id>/`, caseone — `temp/caseone`.
- В `docker-compose.yml` смонтирован writable `./temp:/app/temp`.
- Диалог, загрузка zip/файлов, `incident_id`, уточняющие вопросы в чате.
- Загрузка файлов **не прерывает** диалог (не вызывает LLM без текста пользователя).

**Что ещё осталось от A:** привязка к `tsrag-ollama` / `shared-network`.

## B. Привязка к продукту CaseOne / CaseMap

6. PoC рассчитан на **логи CaseOne**: middleware, `global.log`, `WorkflowTrace.log`, `ClientLogs.log`. Другой продукт (1С, nginx-only, K8s) — эти имена и форматы **не подхватятся сами**.
7. **Долгие запросы (шаг 4)** читаются только из логов, в имени которых есть **`RequestLoggingMiddleware`**, и только строки вида **`GET/POST … /api/... … миллисекунды`**. Другой формат лога → шаг 4 будет пустым, даже если запросы были долгими.
8. **Ошибки (шаг 5)** по умолчанию ищутся **только в файлах, в пути которых есть `global.log`**. Ошибки в других файлах среза **игнорируются**, если не менять настройку API.
9. **Типы ошибок** заданы списком в коде: deadlock, дубликат PK, timeout, «карточка изменилась», обрыв соединения. Текст вне этого списка попадёт в «общая ошибка» или не попадёт вообще.
10. В промптах шагов 0 и 6 прямо написано: инциденты **CaseOne/CaseMap**. Модель настроена на этот домен.
11. Поле **`caseone_path`** проверяется (есть ли папка) и **передаётся в LLM на шаге 0**. **Ни один шаг 1–6 по файлам из caseone не ходит.** Код конфигурации и API-док из caseone **не читаются**.

**Итог:** PoC — разбор **логов CaseOne**, а не «любой системы». Caseone в сессии сейчас почти декоративный для шагов 1–6.

## C. Привязка к одному сценарию («кнопка», сохранение, ProjectTypes)

12. В **примере в форме** — текст про сохранение новой кнопки и 20–30 минут. Это не ломает код, но подталкивает к одному кейсу.
13. В **промпте шага 0** в примерах ключевых слов: `save`, `button`, `PutProjectType`. Модель может подставить их **даже если пользователь про это не писал**.
14. Если LLM не вернула цель разбора, в коде подставляется цель про **«долгое сохранение и ошибки сервера»** — снова сценарий «save».
15. ~~В **README** все примеры — `REN-MSKCASPRO01_2026-04-23`.~~ **Частично исправлено** (пути); доменные примеры вроде `PutProjectType` могут оставаться в тексте кейса.
16. **Классификации `ProjectTypes` vs `Projects` в коде нет.** Есть только группировка по полному URL в шаге 4. Отдельно «это сохранение типа проекта, а не проекта» PoC **не определяет**.

**Итог:** PoC не зашит на один тикет в коде шагов 3–5, но **шаг 0 и UI смещены** под историю с кнопкой и сохранением.

**Что уже сделано по пункту C:**

- Введено поле **`search_keywords`** (генерирует LLM на шаге 0); шаг 3 ищет только по ним, без захардкоженного `symptom_keywords.py`.
- Промпт и guard'ы доработаны (один вопрос за turn, не повторять вопрос, enrich даты/времени из ответов пользователя).

**Что ещё осталось:** нейтральные примеры в промпте/README/UI, fallback «сохранение» в `extractor.py`, классификация ProjectTypes vs Projects.

## D. Жёсткие числа (не из жалобы пользователя)

17. **Медленный запрос** = не короче **5 минут** (300 000 мс). Зашито в кнопке шага 4 в UI. Инцидент «тормозит 30 секунд» на шаге 4 **ничего не покажет**, пока не измените порог в запросе API вручную.
18. **Связка ошибки с долгим запросом** — окно **±90 секунд**. Зашито в UI шага 5.
19. В срез попадает **не больше 100 000 строк**; остальное отбрасывается, в ответе флаг «срез обрезан». Шаги 3–5 видят **только то, что влезло**, не весь инцидент на диске.
20. Одна строка лога в сэмплах обрезается до **2000 символов**.
21. В досье для шага 6 — ограниченное число коротких цитат (порядка **20 фрагментов по ~500 символов**). Это не полные логи.

**Итог:** Пороги «медленно» и «рядом по времени» — **фиксированные**, не из описания инцидента.

---

## E. Что задумывали в алгоритме, но в PoC **нет**

22. **Парсинга `WorkflowTrace.log` нет** — файл может попасть в срез по времени, но отдельного шага «этапы workflow, завис на save» **нет**.
23. **Парсинга client / console логов нет** — нельзя вывести «не браузер» / «обрыв сети на клиенте» из кода PoC.
24. **Исходный код и API-документ не используются** — только текст жалобы и логи.
25. **HITL** (правка заключения человеком) **нет**.

**Итог:** Вывод шага 6 про клиент, браузер и смысл операции **ограничен** тем, что есть в middleware + global + keywords. Без trace/client это **не доказывается**, только предполагается или попадает в «не подтверждено».

---

## F. Фильтр по времени (частая ловушка)

26. Время в логах ищется **подстрокой**, например `2026-04-23 20:`. Если в файле другой формат (`23.04.2026`, UTC без даты, только время) — **срез будет пустым**, хотя логи на диске есть.
27. Часовой пояс **нигде явно не задаётся**. «8 вечера» в промпте переводится в 20:00, но **не привязывается к TZ сервера логов**.

**Итог:** Универсальность по времени зависит от того, совпадает ли формат меток в логах с тем, что сгенерировал шаг 0.

**Дополнение (реализовано позже):** одна метка времени без интервала расширяется до ±10 минут (`expand_single_time_window`); при «достаточно/продолжай» без времени — режим `full_corpus`.

---

## G. Что **универсально** (работает для другого инцидента CaseOne на тех же логах)

28. **Текст жалобы**, **путь к логам**, **дата из имени папки** (если есть `YYYY-MM-DD`) — не привязаны к REN.
29. **Окно времени, patterns, keywords** — из шага 0 (LLM + пользователь), не захардкожены в Python для шага 3.
30. **Шаги 3–5** не ходят на диск повторно — только по срезу из 1–2 (если срез не пустой и не обрезан критично).
31. **Шаг 6** не читает логи с диска — только JSON сводки; другой инцидент = другие факты в сводке, если шаги 0–5 прогнаны.

**Итог:** Для **другого инцидента на том же типе логов CaseOne** (другая дата, другие слова, другой URL) PoC **нормально переиспользуется**, если поправить пути и не полагаться на примеры в промпте.

---

## H. Ошибка в коде (не про инцидент, но вводит в заблуждение)

32. В тексте «ошибок не найдено» всегда пишется **«только global.log»**, потому что в `error_correlation.py` условие `if True` — всегда одна ветка. На самом деле режим зависит от флага запроса, но сообщение **не переключается**.

## Одна фраза без двусмысленности

| Область | Вердикт |
|---------|---------|
| **Не универсально** | Любой продукт, любой формат логов, любой сервер, любой порог «медленно», доказательство «не браузер», caseone/API/код, папки не `D:\RAG` / не `REN` |
| **Универсально в рамках CaseOne** | Новая жалоба → шаг 0 → срез → поиск → заключение, если логи того же типа и формат времени совпадает |
| **Смещено под ваш пример** | UI, README, промпт шага 0, fallback «сохранение», порог 5 минут и 90 секунд в UI |
